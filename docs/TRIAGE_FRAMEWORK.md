# TODO/FIXME Triage Framework

This document defines the categorization and triage process for all TODO and FIXME annotations in the AaaG codebase.

---

## Purpose

TODO and FIXME markers are used to flag incomplete work, known issues, or deferred decisions. This framework ensures that:

1. All markers are categorized by **severity** and **path type**
2. Markers are discoverable and tracked across sprints
3. Critical issues (especially in payment/auth paths) are prioritized
4. The triage report is generated and reviewed during sprint planning

---

## Severity Levels

### Critical

**Definition:** Blocks shipping or poses a security/data-integrity risk.  
**Examples:**
- Unvalidated user input in payment processing
- Missing JWT expiry validation in auth paths
- SQL injection vulnerability
- Unencrypted credential storage

**Action:** Must be resolved before the next release. Escalate to the principal engineer if resolution is unclear.

### High

**Definition:** Degrades user experience or introduces technical debt that compounds risk.  
**Examples:**
- Incomplete error handling in API responses
- Missing test coverage for a core feature
- Deprecated library usage with available replacements
- Known race condition in concurrent code

**Action:** Should be resolved within the current or next sprint. Include in sprint planning.

### Low

**Definition:** Nice-to-have improvements or minor code quality issues.  
**Examples:**
- Refactoring opportunity for readability
- Missing log statement for debugging
- Unused import or variable
- Comment clarification

**Action:** Can be deferred to a future sprint or resolved opportunistically during code review.

---

## Path Types

Path types classify the code domain affected by the TODO/FIXME:

### Payment

**Scope:** Stripe integration, order processing, invoice generation, refund logic.  
**Risk:** High — financial data and transactions are at stake.  
**Triage Rule:** All payment-path TODOs are at least **High** severity. Critical payment issues block shipping.

### Auth

**Scope:** JWT validation, session management, user identity verification, permission checks.  
**Risk:** High — user data and access control are at stake.  
**Triage Rule:** All auth-path TODOs are at least **High** severity. Critical auth issues block shipping.

### Data

**Scope:** Database schema, migrations, data validation, consistency checks.  
**Risk:** Medium-to-High — data integrity and availability are at stake.  
**Triage Rule:** Data-path TODOs should be resolved before schema changes land.

### UI

**Scope:** Frontend components, user flows, accessibility, responsive design.  
**Risk:** Low-to-Medium — user experience is affected, but not data or security.  
**Triage Rule:** UI TODOs can be deferred to a future sprint unless they block a feature.

### Infra

**Scope:** CI/CD, deployment, monitoring, logging, environment configuration.  
**Risk:** Medium — system reliability and observability are at stake.  
**Triage Rule:** Infra TODOs should be resolved before the next release.

### Other

**Scope:** Documentation, build scripts, tooling, or unclear scope.  
**Risk:** Low — does not directly affect user-facing code.  
**Triage Rule:** Other TODOs can be deferred unless they block a feature or release.

## Annotation Format

All TODO and FIXME annotations must follow this format:

```
// TODO(<severity>, <path_type>): <description>
// [Optional: additional context or blocking condition]
```

### Examples

#### Critical Payment Issue

```go
// TODO(critical, payment): validate order total against line items before charging
// Blocked until CI is green and payment path is fully tested.
// See: https://github.com/minions/AaaG/issues/42
```

#### High Auth Issue

```python
# TODO(high, auth): check JWT expiry before processing requests
# Currently accepts expired tokens; this allows unauthorized access.
# Blocked until auth tests are added to CI.
```

#### Low UI Issue

```javascript
// TODO(low, ui): improve mobile responsiveness of dashboard cards
// Can be deferred to Sprint N+2.
```

#### Infra Issue

```bash
# TODO(high, infra): add health-check endpoint to Go API
# Required for Kubernetes liveness probes.
```

## Triage Report Generation

During sprint planning, the triage report is generated by:

1. **Scanning** the codebase for all TODO and FIXME markers
2. **Extracting** severity, path type, and description
3. **Classifying** each marker as:
   - **Resolved** — already fixed or no longer relevant
   - **Tracked** — converted to a GitHub issue with a label
   - **Deferred** — scheduled for a future sprint
   - **Blocking** — must be resolved before the next release
4. **Reporting** the results in `docs/TRIAGE_REPORT.md`

### Scanning Command

```bash
# Find all TODO and FIXME markers in the codebase
grep -r "TODO\|FIXME" --include="*.go" --include="*.py" --include="*.js" --include="*.ts" --include="*.tsx" .
```

### Report Template

See `docs/TRIAGE_REPORT.md` for the current triage report.

## Triage Rules

### Rule 1: Payment and Auth Paths Are High-Risk

Any TODO or FIXME in a payment or auth code path is automatically classified as at least **High** severity. Critical issues in these paths block shipping.

### Rule 2: Blocking Conditions Must Be Documented

If a TODO is blocked by another task (e.g., "blocked until CI is green"), the blocking condition must be stated in the annotation or linked to a GitHub issue.

### Rule 3: Undocumented TODOs Are Not Allowed

All TODO and FIXME markers must follow the annotation format. Markers without severity or path type are flagged during code review and must be clarified or removed.

### Rule 4: Resolved TODOs Must Be Removed

Once a TODO is resolved, the annotation must be removed from the code. Do not leave resolved TODOs in the codebase.

### Rule 5: Deferred TODOs Must Be Tracked

If a TODO is deferred to a future sprint, it must be converted to a GitHub issue with a label (e.g., `tech-debt`, `bug`, `enhancement`) and assigned to a sprint.

---

## Integration with Sprint Planning

During sprint planning:

1. The triage report is reviewed by the principal engineer and manager
2. Critical and high-severity TODOs are prioritized for the current sprint
3. Low-severity TODOs are deferred or resolved opportunistically
4. Blocking conditions are assessed and sequenced
5. The triage report is updated and committed to the repo
